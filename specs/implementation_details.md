# 実装詳細仕様書: dialy

本ドキュメントは、要件定義およびDB設計を具体的にコードへ落とし込むための実装方針を定義したものである。

## 1. セキュリティ実装方針 (Authentication & Authorization)
- **採用技術**: Spring Security
- **認証方式**: セッションベースの認証
- **詳細**:
    - `UserDetailsService` を実装し、DB（`users` テーブル）からユーザー情報を取得する。
    - パスワードは `BCryptPasswordEncoder` を使用してハッシュ化して保存・照合する。
    - 認証済みユーザーのみが日記の操作（作成・編集・削除）を行えるよう、URLベースのアクセス制限を設定する。

## 2. UI/CSS フレームワーク
- **採用技術**: Bootstrap (最新の安定版)
- **詳細**:
    - Thymeleaf テンプレート内で Bootstrap の CDN または静的リソースを利用する。
    - レスポンシブデザインを適用し、PCおよびスマートフォン双方で閲覧可能とする。
    - 標準的なコンポーネント（Navbar, Card, Modal, Form等）を使用して効率的に構築する。

## 3. バリデーションルール (Bean Validation)
以下の制約を `jakarta.validation.constraints` 等を用いて実装する。

| 項目 | 制約内容 | 備考 |
| :--- | :--- | :--- |
| `username` | 必須, 2〜20文字 | 空文字不可 |
| `email` | 必須, メール形式 | 正しいメールフォーマットであること |
| `password` | 必須, 8文字以上 | セキュリティ強度の確保 |
| `diary.title` | 必須, 最大100文字 | |
| `diary.content` | 必須, 最大10,000文字 | |
| `diary.entry_date` | 必須 | 日付が正しく指定されていること |

## 4. エラーハンドリング方針
- **共通処理**: `@ControllerAdvice` を導入し、アプリケーション全体で発生する例外を共通形式で捕捉する。
- **ユーザーへの通知**:
    - バリデーションエラー時は、BindingResult を介して入力フォームの該当箇所にエラーメッセージを表示し、元のページへリダイレクトさせる。
    - ビジネスロジック上のエラー（例：存在しない日記へのアクセス）は、適切なカスタム例外をスローし、専用のエラーメッセージを表示する。
- **エラーページ**:
    - 404 (Not Found) および 500 (Internal Server Error) のためのカスタムHTMLページを用意し、ユーザー体験を損なわないようにする。
